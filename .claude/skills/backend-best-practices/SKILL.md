This skill is about the code *inside* a module. For module boundary mechanics see `backend-spring-modulith`; for the reasoning behind boundary/event/transaction/security rules see `doc/architecture/RULES.md`; for test-tier selection see `testing-strategy`. Don't duplicate those here — this skill assumes you've read `RULES.md` §1–6 and is about how to write the Java that satisfies those rules well.

## 1. DDD-informed entity design

**Rich entities, not anemic ones.** `Itinerary.markAsQuotation()` (throws `IllegalStateException` unless `status == DRAFT`) is the reference pattern — state-transition rules live as methods on the entity that owns the state, not as `if` checks scattered across every service method that might trigger the transition. Why this matters more than usual here: ADREN's core entities (`Itinerary`, `Booking`, `Quotation`, `Package`, `AdCampaign`) all have PRD-defined state machines (Draft→Quotation→Booked→Cancelled; PendingApproval→PendingPolicyReview→Live→Paused/Rejected) with edge cases the PRD explicitly calls out (§23) — if the invariant is enforced in one place, a new call site can't reintroduce a bug the PRD already anticipated. When adding a new state-bearing entity:

- No public setters for state fields. Mutation happens through named methods expressing the *business* operation (`confirm()`, `cancel()`, `applyPartialRefund(Money)`), never `setStatus(BOOKED)`.
- The entity throws (`IllegalStateException` for an invalid transition, a specific checked/unchecked domain exception for a business-rule violation — see §2) rather than silently no-op'ing on an invalid transition. A silent no-op here is how PRD §23.1 Edge Case #3 ("traveler count changes after Quotation but before booking — price must be recalculated, not carried over") turns into a real bug: if `Quotation.recalculate()` silently did nothing when called on a non-recalculable state instead of throwing, the stale-price bug would ship without any test failing.
- Value objects (`Money`, and anything with the same "immutable, self-validating, no identity" shape) are Java `record`s, matching `shared.Money`. Don't reach for a mutable class with getters/setters for something that's conceptually a value.

**Aggregate boundaries follow the module boundaries**, not the other way around. `Itinerary` and its line items are one aggregate inside `booking`; a `Booking`'s `supplier_booking_refs[]` (PRD §20.8) references suppliers by ID, not by holding a live reference into `supplier`'s entities — cross-module references are always by ID (§4.1 of `RULES.md`), which happens to also be correct DDD aggregate design (aggregates reference other aggregates by identity, not by object graph).

## 2. Exception handling conventions

- **Distinguish three exception categories explicitly**, because they get handled differently at the `@ControllerAdvice` boundary (`RULES.md` §3.3):
  1. **Domain/business-rule violations** — expected, user-facing, mapped to 4xx (`RateExpiredException`, `CreditLimitExceededException`, `InvalidStateTransitionException`). Model as unchecked exceptions extending a per-module base (e.g. `BookingDomainException`) so a module's `@ControllerAdvice` can catch its own family without a giant instanceof chain.
  2. **Supplier integration failures** — expected-but-external, need the differentiated user messaging the PRD spells out per supplier (§10.2's per-supplier error tables: "temporarily unavailable" vs. "rate expired" vs. "no coverage at this route" are three *different* user messages, not one generic "supplier error"). Model each distinct supplier failure mode as its own exception type (`HotelbedsClient.HotelbedsRateExpiredException` is the existing example) — don't collapse them into a single `SupplierException` with a string reason, because that pushes the message-selection logic into a fragile string-matching `switch` at the call site instead of the type system doing it.
  3. **Programming errors / invariant violations** — unexpected, 5xx, should alert (not just log) in production. `NullPointerException`, an assertion failure, an unmapped enum case in a `switch`. Never catch these broadly and convert them to a generic error response that looks the same as a 4xx to the client — a Consultant should never see "something went wrong" for the same class of problem as "your rate expired," because the former means Adren has a bug and the latter means normal business flow.

- **Never swallow an exception silently without a documented reason.** `SupplierAggregationService.searchHotels`'s current bare `catch (Exception e) { /* swallow */ }` is *correct in intent* (per PRD §9.4: one supplier's failure must not fail the whole search) but is currently swallowing too broadly — it should catch the specific exception types §10.2 defines as "exclude from this cycle" cases (timeout, no-availability) and let genuinely unexpected exceptions (a `NullPointerException` from a mapping bug) propagate or at minimum get logged at ERROR with full context, not silently disappear at the same log level as an expected timeout. A swallowed `NullPointerException` in a supplier mapper is a silent data-quality bug, not resilience.
- **Exception messages that reach the client never include supplier-internal details** (raw supplier error codes, stack traces, internal IDs) — map to the PRD §10.2 user-facing message table explicitly. Log the raw supplier error server-side (with the correlation ID from `RULES.md` §6.1) for debugging; never forward it verbatim to a Consultant.
- **Don't use exceptions for expected control flow within a single method** — "no availability" from a supplier is not exceptional, it's a normal search outcome; model it as an empty result / a sentinel in the search result type, not a thrown-and-caught exception inside the same method that produced it. Reserve exceptions for genuinely exceptional conditions or for signaling failure *across* a method/module boundary (a supplier client throwing so its caller's catch block can apply the §10.2 fallback behavior is the right use).

## 3. Concurrency rules

**Why this needs its own section beyond "Spring handles it":** PRD §23.1 Edge Case #1 is explicit — two Users under the same Consultant booking the last available inventory unit simultaneously must have the second attempt fail gracefully, not double-book. This is a real concurrency problem, not a hypothetical.

- **Optimistic locking (`@Version`) on any entity representing a finite, contended resource** — inventory holds, wallet balances, credit limits. A `@Version` field on `Itinerary`/`Booking`/wallet-ledger-adjacent entities turns the "two Users book the last unit" race into a detectable `OptimisticLockException` at the second commit, which the service layer maps to the PRD's required "no longer available" message (§23.1 Edge Case #1) — rather than a silent double-write.
- **Never rely on a read-then-write pattern across two separate transactions for anything contended.** "Check wallet balance, then in a second transaction debit it" has a race window; the check-and-debit must be one transactional unit, ideally backed by a DB-level constraint (a `CHECK` constraint or a conditional `UPDATE ... WHERE balance >= amount` with the row-count checked) as the actual enforcement, with the application-level check as the fast-path/UX layer, not the sole guarantee. PRD §24.4 explicitly requires wallet ledger writes to be "atomic and idempotent (no double-debit on retry)" — that's a database-transaction-and-constraint requirement, not something achievable purely in application code with a `if (balance > amount)` check.
- **`@ApplicationModuleListener`-based side effects run asynchronously on a separate thread pool** (documented Modulith behavior, confirmed by `notification`'s listener). Any state a listener reads must be re-fetched fresh inside the listener (don't pass a JPA entity reference across the async boundary — it's detached and potentially stale by the time the listener thread reads it; pass IDs and primitive/record data in the event payload, exactly as `BookingConfirmedEvent` already does).
- **Rate-limiting for outbound supplier calls uses a shared, thread-safe limiter per supplier** — a token-bucket implementation (e.g. Resilience4j's `RateLimiter`, or `Bucket4j`) held as a singleton bean per supplier client, not a per-request or per-thread-local counter, since concurrent search requests from different Consultants share the same underlying supplier rate-limit budget (PRD §10.2 is explicit that Hotelbeds/STUBA/TBO/Mystifly limits are independent per-supplier budgets, not one global budget — Mystifly in particular "needs its own dedicated rate-limit bucket, separate from hotel suppliers" per §10.2.4).

## 4. Dependency injection conventions

- **Constructor injection only** — no field injection (`@Autowired` on a field), no setter injection. Already the pattern in every existing service/controller (`BookingServiceImpl`, `ItineraryController`, `SupplierAggregationService`, `HotelbedsClient` all constructor-inject). This isn't just style: constructor injection makes a class's dependencies visible in its signature (so a reviewer sees coupling growth immediately — a constructor gaining a 6th parameter is a visible code smell that field injection hides), and it makes unit testing without a Spring context possible (`BookingServiceImplTest` mocking `ItineraryRepository`/`ApplicationEventPublisher` directly depends on this).
- **Depend on the module's `Api` interface, never the `*ServiceImpl`**, even from within tests that could technically reach the impl — `ItineraryController` depending on `BookingApi` rather than `BookingServiceImpl` is correct and must stay correct as more controllers/listeners are added. If a consumer needs something the `Api` interface doesn't expose, that's a signal to grow the `Api` interface, not to reach past it.
- **A constructor with more than ~4-5 dependencies is a decomposition signal**, not a "use a facade object" signal — `SupplierAggregationService` is explicitly planned to grow to `HotelbedsClient` + `StubaClient` + `TboClient` + `LocalDmcRepository` + `ByosClient` (per its own TODO comments); when that lands, consider whether "aggregate across N supplier clients" is better modeled as an injected `List<SupplierClient>` (one common interface, Spring collects all implementing beans) than five named constructor parameters — this also means adding a tenth supplier later doesn't require editing the aggregator's constructor at all, just registering a new bean.
- **No static service locators, no `ApplicationContext.getBean(...)` calls outside framework-integration code.** If a class needs a collaborator, it's a constructor parameter.

## 5. Performance patterns

### N+1 avoidance
- Every `findBy...` repository method that will be called in a loop (e.g., resolving supplier names for a list of line items) needs either a fetch-joined query (`@Query("... join fetch ...")`) or a batch method (`findAllById`/`findByXIn`) from the start — don't ship the naive per-item lookup "for now." Given search aggregation fans out across up to 7 live suppliers plus Local DMC plus BYOS per PRD §10.1, any per-result-item lazy load multiplies immediately.
- Prefer projections (a `record`/interface-based Spring Data projection) over fetching full entities when a list endpoint only needs a few fields — a paginated `findBookingsByConsultant` (per `RULES.md` §3.4's reconciliation item) should very likely return a projection, not full `Booking` aggregates with all their line-item collections eagerly or lazily attached.
- Set `spring.jpa.open-in-view: false` and keep it that way (already configured) — this forces lazy-loading exceptions to surface at the service layer during development instead of silently working via an open Hibernate session in the view layer, which is exactly the mechanism that hides N+1 problems until production load reveals them.

### Pagination
- Covered at the API contract level in `RULES.md` §3.4 — the implementation-level rule is: never fetch an unbounded collection from the database to paginate it in memory. Use `Pageable` all the way down to the repository query (`Page<Booking> findByConsultantId(UUID consultantId, Pageable pageable)`), not `findByConsultantId(consultantId).stream().skip(...).limit(...)`.

### Caching strategy
- **Cache static/slow-changing content, never live pricing or availability.** PRD §10.5's split (real-time search/pricing vs. scheduled sync for static content — images, descriptions, amenities) maps directly onto what's cacheable: hotel content, activity descriptions, cruise ship/deck-plan images are cache candidates (a scheduled-refresh cache, e.g. `@Cacheable` backed by a TTL matching the nightly/weekly sync cadence per supplier from §10.2). Rate/availability responses are **never** cached beyond the single search request's own in-memory aggregation — caching a rate risks exactly the `RATE_STALE` scenario §10.2.1 already has to handle for uncached data; caching would make it worse, not better.
- **Supplier-content caches are keyed to include the currency/market where content varies by market presentation**, not just by supplier+property ID, if/when localized content (multi-language per PRD §13.3) is introduced — a cache key collision across languages is a subtle, easy-to-miss bug class here.
- **The AI governance module must not cache AI suggestions across different underlying inventory states.** PRD §11.2's grounded-generation principle ("AI may only select from live, supplier-confirmed inventory") means an AI itinerary suggestion is only valid against the availability snapshot it was generated from — caching/reusing a suggestion across requests risks silently violating the "re-validate at booking time" requirement (§11.3) if the cache TTL outlives the underlying rate's validity window. If AI suggestion caching is ever introduced for latency reasons, the cache must be invalidated by the same staleness signal that would trigger re-validation, not a fixed TTL chosen independently of supplier rate volatility.

## 6. Supplier-aggregation module specifics (`supplier`)

- **One client class per supplier, one failure-isolation boundary per client** — `HotelbedsClient` living in its own `internal.hotelbeds` sub-package (separate from the generic aggregation logic in `internal`) is the right shape; replicate it as `internal.stuba.StubaClient`, `internal.tbo.TboClient`, etc. Don't build one generic "SupplierClient" class with a `switch (supplierId)` inside — PRD §10.2 makes clear each supplier has materially different auth (signed hash vs. session token vs. API key), request/response shapes, and error semantics; a shared class encourages leaky abstractions across those differences.
- **Circuit breaker per supplier client, independently configured** — PRD §24.2 requires this explicitly ("each supplier integration isolated behind a circuit breaker — one supplier's downtime must not degrade search latency for the others"). This is not yet implemented (no Resilience4j dependency exists in `build.gradle.kts` today) — when added, wrap each client's outbound call individually (`@CircuitBreaker(name = "hotelbeds")` on `HotelbedsClient.search`, a separately named/configured breaker per supplier, not one global breaker), so Hotelbeds tripping doesn't affect the STUBA or TBO breaker's state.
- **Aggregation fan-out is parallel, with a bounded overall timeout**, not sequential — search across 7+ suppliers sequentially would blow the "low single-digit seconds" search SLA (PRD §24.1) immediately. Use `CompletableFuture`s (or WebFlux's reactive composition, given `spring-boot-starter-webflux` is already a dependency and `HotelbedsClient` already uses `WebClient`) to fan out and join with a timeout, treating a per-supplier timeout as "exclude this supplier's results," never as "fail the whole search" (matches §10.2's per-supplier error table: timeout → "showing results from other suppliers," not a blank screen).
- **`TraceId`/session-token state (TBO, STUBA) is itinerary-draft-scoped, not request-scoped.** PRD §10.2.3 is explicit that TBO's `TraceId` must be stored against the in-progress itinerary draft and reused across the whole search-to-booking window — this means the supplier module needs a place to persist that session token tied to the draft itinerary (likely a small `internal` entity or a Redis-backed session keyed by `itinerary_id`), not just held in a request-scoped bean that dies at the end of the search HTTP call.
- **Normalize suppliers to a common result shape before returning from `SupplierSearchApi`** — `SupplierSearchResult` already does this for hotels (`supplierRateId`, `propertyName`, `roomType`, `netRate` as `Money`) regardless of which of the three hotel suppliers produced it. Extend the same discipline to the other product categories as their clients are built (flights, transfers, cruise, activities each get their own normalized result record per PRD §20.2–20.6), and keep supplier-specific fields (TBO's `TraceId`, Hotelbeds' opaque `rateKey`) inside the client/internal layer — never leak a supplier-specific field name into the public `Api` surface.
- **BYOS credential-source-agnosticism (PRD §10.2.9) is a dependency-injection concern, not a branching concern.** The rule "the same Hotelbeds integration code path should work whether credentials are Adren's own or a Consultant's BYOS credentials" means `HotelbedsClient` should accept a credential set as a parameter/injected-per-call value (resolved once, upstream, by whichever credential source is active for the current request's tenant — see `RULES.md` §5.3 for how BYOS credentials must be stored/read), not contain an `if (isByos) {...} else {...}` branch inside the client itself. Branching on credential source inside the client is exactly the coupling this requirement is designed to avoid.

## 7. AI-governance module specifics (`ai`)

- **Every AI call is logged before the response is used, not after** — PRD §11.2/§24.3's "100% of AI suggestions logged, no sampling" requirement means the audit write and the AI response's use in the itinerary builder should not be separable — if the audit-log write fails, the suggestion should not silently be usable/displayed, because that would create suggestions that exist in the UI with no corresponding audit trail, defeating the purpose. Treat the audit-log write as the transactional gate; wrap it so a failed audit write surfaces as an error to the caller rather than being a best-effort side channel.
- **AI suggestions carry their grounding data explicitly in the response type**, not just in the log — PRD §11.3's acceptance criterion ("every line item shows supplier source and live availability before approval") means the `Api` response DTO for an AI suggestion must include `supplierId`/availability-as-of-timestamp per line item as first-class fields, not something the frontend has to separately fetch or infer.
- **Model failure/empty-result as a first-class response state, never an exception the caller has to interpret.** "AI states inability rather than substituting" (PRD §11.2 principle 4) means "no valid suggestion" is a legitimate, well-typed response value (e.g. a sealed/enum-discriminated result: `Suggested(items)` vs. `NoViableSuggestion(reason)`), not a thrown exception that risks being caught-and-papered-over by a generic error handler somewhere up the call stack into a misleading message.
- **Never let a retry/timeout on the underlying model call (Groq, per the existing `adren.ai.groq` config) silently produce a different suggestion than what gets logged.** If the client-side call is retried, log each attempt distinctly (not just the final one) — an audit trail that only records the successful attempt after two failed ones loses information a governance review might need, and could misrepresent what the AI "actually said" across the retries.

## 8. Backend checklist addendum

Beyond `RULES.md` §8's general PR checklist, for changes inside `supplier`/`ai` specifically:

- [ ] New supplier client is its own class in its own `internal.<supplier>` sub-package, not a branch inside a shared class.
- [ ] New supplier call is wrapped for the specific failure modes its PRD §10.2 subsection defines (not a bare catch-all).
- [ ] New supplier call participates in the shared aggregation fan-out with a bounded timeout, not sequential blocking.
- [ ] Any contended-resource write (inventory hold, wallet debit) has an optimistic-lock or DB-constraint guarantee, not just an application-level check.
- [ ] Any AI suggestion path writes its audit log transactionally before/with the response, not as a fire-and-forget side effect.
- [ ] Any new list-returning repository method takes `Pageable` from the start.
