# ADREN TRAVEL — Architecture Narrative

**Audience:** a new engineer or stakeholder who needs the *reasoning* behind the system's shape, not just its structure. For structure diagrams, see the PlantUML output under `doc/architecture/` once generated (`backend/README.md` — "Generating module documentation"; **as of this writing that directory contains only `RULES.md`, no diagrams have been generated yet** — see §9 below). For the enforceable engineering rules this narrative explains the "why" behind, see `doc/architecture/RULES.md`. For the white-label design-system rules, see `doc/DESIGN.md`.

---

## 1. System context

ADREN TRAVEL is a B2B SaaS platform that lets travel Consultants — independent agents and small agencies, based first in India and expanding to Australia, the UK, the USA, Dubai/UAE, and Denmark — build, price, and sell multi-product trip itineraries (hotels, flights, transfers, cruises, activities) under their own branded storefront, without needing their own supplier contracts, booking technology, or marketing capability. Adren supplies the shared infrastructure — supplier aggregation, AI-assisted itinerary building, payments/markup/wallet, white-label branding, and even managed Meta ad campaigns — and each Consultant's own end travelers experience it as that Consultant's business, not Adren's (PRD §0, §1, Part 3).

## 2. Why a modular monolith (Spring Modulith), not microservices

The backend is one Spring Boot deployable containing ten bounded contexts (`booking`, `supplier`, `ai`, `payments`, `whitelabel`, `ads`, `notification`, `compliance`, `security`, `shared`), with Spring Modulith enforcing module boundaries at build/test time rather than a network boundary enforcing them at runtime (`RULES.md` §1's opening paragraph states this framing directly).

The tradeoff, made explicit: at this project's stage — a small team building nine-plus bounded contexts concurrently against an MVP timeline — a microservices split would front-load costs (per-service deployment pipelines, network-boundary observability, distributed-transaction handling for what are today single-transaction flows) before any single module has demonstrated it actually needs independent scaling or an independent release cadence. A monolith done *carelessly* degenerates into hidden coupling with none of a monolith's simplicity benefits; Spring Modulith is specifically the mechanism that prevents that degeneration without paying the microservices cost early — `ApplicationModules.verify()` (wired into `./gradlew check`) fails the build on a cross-module `.internal` import, giving compile/test-time enforcement of boundaries a plain multi-package monolith wouldn't have (`RULES.md` §1.1–§1.2).

**What would trigger reconsidering this decision:** `RULES.md` §1 already names the realistic candidates — `payments` and `compliance` are flagged as modules where "a hard split... is a realistic future, not a hypothetical" given the six-jurisdiction regulatory load this platform carries and payments' distinct security/PCI posture (§24.4). A second concrete trigger, evidenced by the production story catalogue rather than asserted here: `PERF-03` (load-test supplier fan-out under partial supplier degradation) and the per-supplier circuit-breaker requirement (§24.2) point at `supplier` as a module that may need to scale/fail independently of the rest of the platform once real supplier traffic volumes exist. The decision to split would be made per-module, when one of these shows measured pressure — not as a wholesale re-architecture.

## 3. Why event-driven internally, and where a direct call was chosen instead

`RULES.md` §2.1 states the decision rule this codebase already follows: a **direct call** through a module's `Api` interface when the caller needs a synchronous result or must know about failure before its own transaction commits; a **domain event** when the action is a side effect other modules react to independently, and the originating transaction's success must not depend on that side effect succeeding.

The realized pattern is `BookingConfirmedEvent`: booking confirmation publishes it, and `notification`'s `@ApplicationModuleListener` reacts asynchronously (after the originating transaction commits) — so a slow or failing WhatsApp/email send can never block or fail a booking confirmation. The same event is the intended fan-out point for `payments` and eventually `ads` (PRD §23.5 Edge Case #11 — a package price change pausing a live ad campaign) without `booking` ever needing to know `ads` exists (`RULES.md` §2.1). That's the benefit realized: decoupled, additive fan-out.

The complexity cost is real, not hand-waved: Spring Modulith's JPA event publication registry gives **at-least-once** delivery, not exactly-once, so every listener must be independently idempotent (`RULES.md` §2.2) — natural idempotency (state-check-then-transition) is preferred over an infrastructure dedup cache, because an in-memory cache doesn't survive the exact crash-between-steps scenario the outbox pattern exists to survive. This is deliberately called out as unfinished work: `BookingNotificationListener`'s body is currently a `// TODO` stub, and `RULES.md` §2.2 is explicit that idempotency must ship with the *first* real implementation, not be bolted on after.

**Where a direct call was chosen instead:** `ItineraryController` → `BookingApi.saveAsQuotation` is a direct, synchronous call — the HTTP client needs the result before the controller can respond, which is exactly `RULES.md` §2.1's first direct-call criterion. The same reasoning extends to compliance gates that must block an action before it commits (e.g., the UK ATOL disclosure gate blocking Package publish, PRD §17.2/§22.3) — a blocking precondition cannot be an async event, because the caller needs to know about failure *before* proceeding, not react to it afterward.

## 4. The module map

Status column reproduced from `backend/README.md`'s module table (not re-derived) — cross-reference it directly for the authoritative status, and see §8 below for one discrepancy found between that table and the actual repository.

- **`booking`** (PRD §9) — the core itinerary → quotation → package → booking lifecycle. **Reference implementation** — the pattern every other module follows.
- **`supplier`** (PRD §10) — normalizes and aggregates inventory from Hotelbeds, STUBA, TBO, Mystifly, Transferz, Widgety, HBActivities, plus Local DMC and BYOS sources behind one `SupplierSearchApi`. **Reference implementation**, but only for one supplier — see §8's risk note.
- **`ai`** (PRD §11) — grounded itinerary generation and governance (mandatory human approval, 100%-logged audit trail, explicit failure states — never fabricate). **Package-info stub** — no implementation exists yet beyond the module boundary declaration.
- **`payments`** (PRD §12) — markup/commission engine, multi-currency FX buffer, wallet/credit-limit, Stripe. **Package-info stub.**
- **`whitelabel`** (PRD §13) — Consultant branding/domain configuration; this is the backend counterpart to `doc/DESIGN.md`'s Layer 2 architecture (see §7 below). **Package-info stub.**
- **`ads`** (PRD §14) — Meta ad account provisioning, campaign lifecycle, AI creative generation. **Package-info stub.**
- **`notification`** (PRD §15) — dispatches booking/payment/cancellation/campaign notifications across email plus a region-configurable secondary channel. **Reference event listener implemented** — meaning the `@ApplicationModuleListener` wiring/pattern exists as the reference shape; its actual dispatch body is a TODO stub (see §3 above), which is a precision worth holding onto rather than reading "implemented" as "complete."
- **`compliance`** (PRD §17) — per-market KYC/licensing rules, GST/TCS, UK TOMS VAT, ATOL enforcement, GDPR/DPDP data-subject rights. **Package-info stub.**
- **`shared`** — the one deliberately `OPEN` module (`RULES.md` §1.3), holding `Money` and `CurrencyCode` — stateless value types with zero business logic by design. **Complete.**
- **`security`** — authentication/authorization contract: `AdrenPrincipal` (userId/role/consultantId), `Role`, `CurrentPrincipal`, `CapabilityGrantService`. **Not listed in `backend/README.md`'s module table at all** despite being real, substantial code (219 lines) — see §8.

**Dependency shape — events vs. direct calls:** `booking` → `notification` is the one wired event today (`BookingConfirmedEvent`); `payments` and `ads` are designed to join that same fan-out once built (§3). Compliance gates (ATOL, GST/TCS, KYC) are direct calls by necessity — they block a transaction rather than react to one. `security` is consulted synchronously by every module via method-level `@PreAuthorize` on `Api` interfaces (`RULES.md` §5.1), which is a direct-call relationship by construction, not an event. `whitelabel`'s branding-change propagation (FND-07) is designed around a short-TTL cache + invalidation event internally, distinct from the cross-module domain-event pattern above.

## 5. Why LocalStack for MVP, and what changes in production

Local/CI development runs Postgres + LocalStack (S3, SQS, SNS, Secrets Manager) via `backend/docker-compose.yml`, so every engineer and CI run gets a realistic AWS-shaped environment without touching real cloud infrastructure or cost. The full replace-with-real-AWS path — Secrets Manager + rotation Lambdas, S3 + encryption-at-rest, SQS/SNS at production fan-out scale, production Postgres with backup/PITR — is the **Production Infrastructure (PINF)** epic; see `doc/user-stories/production-stories.md`'s Production Infrastructure section (PINF-01 through PINF-11) rather than this document for the item-by-item scope, since that catalogue is the maintained source for it.

## 6. The white-label/multi-tenant architecture

Every Consultant's data — itineraries, wallets, traveler PII, BYOS credentials — must be invisible to every other Consultant. `RULES.md` §5.2 states the core principle this codebase is built around: a UUID primary key is not an access-control mechanism, and Broken Object Level Authorization (OWASP API1:2023) is named as the realistic top risk for this specific platform. Enforcement is designed as tenant-aware-principal-plus-service-layer-check, not a query filter bolted on afterward — `AdrenPrincipal` carries `consultantId`, and `CurrentPrincipal.resolveTenantScope()` is the one place a service method should resolve "which tenant am I actually allowed to query," rejecting any `CONSULTANT`/`USER` principal that doesn't match the resource's owning tenant (`RULES.md` §5.2; the current gap this closes is tracked as `FND-03`). BYOS credentials go a step further than itinerary-level isolation — they're row-level, per-Consultant encrypted secrets (KMS envelope encryption) rather than a shared Secrets-Manager-by-supplier-name entry, specifically because the blast radius of a BYOS leak (another Consultant's live supplier account) is worse than an itinerary leak (`RULES.md` §5.3; tracked as `FND-12`).

## 7. The Layer 1/Layer 2 design system split

`doc/DESIGN.md` §0 and §3 define the frontend counterpart to the tenant-isolation problem above: Adren's own product chrome (Super Admin Console, Consultant Dashboard, Itinerary Builder — "Layer 1") is a fixed design system never influenced by tenant input, while the Consultant's storefront/quotation/voucher surfaces an End Traveler actually sees ("Layer 2") are themed at runtime from four Consultant-supplied fields (logo, background image, two text colors) that are treated as untrusted, unvalidated input until proven otherwise. The reason this needed architecting up front rather than bolted on: those four fields are picked by a travel agent, not a designer, and DESIGN.md's contrast-safety algorithm (§3.3) is the mechanism that guarantees a Consultant's arbitrary color choice can't make an error message or a price illegible — see DESIGN.md directly for the algorithm itself rather than this document.

## 8. Known architectural risks and gaps, stated honestly

Pulled from the verified current state of the repo, not a generic risk list:

- **Only one supplier has a real client.** `supplier/internal/` contains exactly one integration package, `hotelbeds/` — STUBA, TBO, Mystifly, Transferz, Widgety, and HBActivities are all still search-only entries in PRD §10.1's table with no corresponding code yet, despite `backend/README.md` describing the module as a "reference implementation" (accurate for Hotelbeds; the other six suppliers are unscoped work, tracked one-per-supplier in the production `SUP-*` stories).
- **The AI module is Groq-only, with no production failover.** `build.gradle.kts` wires a plain `WebClient` against Groq's OpenAI-compatible REST API with no provider-abstraction layer; `ai/internal/` is currently empty (package-info-only). The swappable-provider abstraction and failover path are explicitly deferred to production (`LLM-02`, `LLM-09`) — meaning until that lands, a Groq outage has no fallback in the MVP build.
- **No circuit breakers exist yet.** `RULES.md` §9 item 9 already tracks this: Resilience4j is not a build dependency, `SupplierAggregationService` currently relies on bare try/catch, and this is a named NFR gap against PRD §24.2 (one supplier's downtime must not degrade search latency for the others) — not yet true today.
- **No authentication is actually enforced**, despite the `security` module's data-model contract existing (see §9's discrepancy #2 below for the precise nuance) — every controller is reachable unauthenticated today. `RULES.md` §5 is unambiguous that this must close before a second/third controller is built against real Consultant/traveler data.
- **Transactional boundaries are incomplete on the one real write path.** `BookingServiceImpl` has no `@Transactional` anywhere yet (`RULES.md` §4.3) — the outbox pattern's atomicity guarantee (entity save + event publish in one transaction) doesn't hold until this lands, tracked as the single highest-priority reconciliation item in `RULES.md`.
- **`BookingConfirmedEvent` still carries decomposed `BigDecimal totalSellPrice` + `CurrencyCode currency`** instead of the shared `Money` type (`RULES.md` §2.3) — the exact anti-pattern the Money rule (§4.4) exists to prevent, currently harmless only because nothing consumes the event for real yet.
- ~~The story-dependency graph contains an unresolved cycle~~ **Fixed** — see §9 below; this was a data-integrity gap in the planning artifacts, not the codebase, that blocked mechanically deriving a strict build order for a cluster of 24 stories centered on `BOK-13`/`BOK-14`. Corrected at the source (the story files' `dependencies:` frontmatter); the full 225-story graph is now re-verified acyclic.

## 9. Discrepancies found while writing this document — flagged, not silently resolved

- **`backend/README.md`'s module table omits the `security` module entirely.** It exists (`AdrenPrincipal`, `Role`, `CurrentPrincipal`, `CapabilityGrantService` + impl, 219 lines total) and is referenced throughout `RULES.md` §5, but doesn't appear as a row in the README's module-status table alongside the other nine modules. The table should either add it (status: contract/scaffold — see next point) or the README should explain why it's omitted.
- **`RULES.md` §5's literal claim is now partly stale.** It states "there is no authentication or authorization mechanism in the codebase at all today — no Spring Security dependency, no filter chain, no principal." As of this writing: `spring-boot-starter-security` **is** a `build.gradle.kts` dependency, and `AdrenPrincipal`/`Role`/`CurrentPrincipal` **do** exist as real types. What's actually still missing — verified by grep, not assumed — is any `SecurityFilterChain`, `@EnableWebSecurity` configuration, JWT signing/parsing, or a single `@PreAuthorize` usage anywhere in the codebase; `CurrentPrincipal.get()` would throw today on every call because nothing ever populates `SecurityContextHolder`. The practical risk RULES.md is warning about (no request is actually authenticated or authorized) is still 100% accurate — only the literal "no dependency, no principal" phrasing is out of date. Whoever next edits `RULES.md` §5 should update the claim to match, per this document's own §0 rule ("update it in the same PR that changes a convention").
- **The mock-phase story dependency graph had a genuine cycle — now fixed at the source.** `BOK-13` ("Build the Direct Booking & Payment flow") listed `BOK-14` ("Capture Traveler Profile details") as a dependency, and `BOK-14` listed `BOK-13` right back. Separately, `BOK-13` → `FIN-08` → `FIN-07` → `BOK-13` was a second, independent 3-cycle sharing the `BOK-13` node, pulling 24 stories into one strongly-connected component that couldn't be topologically sorted as authored. Fixed by removing the two back-edges that were the actual errors: `BOK-14`'s `dependencies:` is now `[]` (it's self-contained; `BOK-13`'s screen consumes it, not the other way around — `BOK-13` correctly still lists `BOK-14`), and `BOK-13`'s `dependencies:` no longer lists `FIN-08` (`FIN-07`'s own developer notes already stated it's "invoked from BOK-13's booking flow," so the prerequisite direction runs `BOK-13` → `FIN-07` → `FIN-08`, not the reverse). Full reasoning in `doc/phases.md` §4 and inline in both story files; the 225-story graph is re-verified acyclic.
- **Four of PRD §19's eight "Open Items for Business Confirmation" have no corresponding story-level flag anywhere** in the 225-file story catalogue (verified by full-text search, not sampling): exact commission tiers, Local DMC vetting ownership (Adren vs. Consultant self-certification), sub-agent hierarchy depth, and the six-timezone support coverage model. The other four items each have one or more explicit `⚠️ NEEDS CLARIFICATION` story flags. See `doc/phases.md` §6 for the full consolidated list and this gap's treatment.
